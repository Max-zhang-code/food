# 智能点菜小程序 Brainstorm

> Stage 0 | 2026-07-01 | 下一步：cs-feat-design

## 想做什么、为什么

核心问题："不知道吃什么"时，通过「我有什么食材」反查「能做什么菜」，降低决策门槛。

**关键约束与决策**：
- B 规模：多人共享（家庭/小团队协作），需要后端 + 数据库，但功能边界清楚
- 微信一键登录，不做独立账号体系
- 暂不做支付，只生成订单

## 考虑过的方向

### 方向 A：uni-app + Vue 3 + 微信云开发（选定）

- 描述：Vue 3 标准写法，uni-app 在编译阶段翻译成微信小程序代码。CLI + VS Code + Vite 开发，npm run dev:mp-weixin 编译预览
- 价值：Vue 3 生态全保留，开发体验好；云端零运维，小规模基本免费；万一未来跨端成本低
- 代价：多一层编译抽象，偶有兼容适配延迟

### 方向 B：Taro + Vue 3（否决）

- Vue 3 支持偏弱，Taro 团队主业在 React 侧，更新节奏慢
- 结论：生态和社区不如 uni-app 稳

### 方向 C：原生微信小程序（否决）

- 没有 Vue 生态，用 WXML/WXS —— 放弃 Composition API、Pinia、.vue 单文件组件整个工具链
- 对于有 Vue 3 基础的开发者，迁移成本高于跨端编译的风险

## 选定方向与遗留问题

**核心行为**：
1. 微信一键登录进入小程序
2. 浏览/选择食材 → 系统匹配可做菜品 → 选菜加入购物车 → 生成订单
3. 查看历史订单记录
4. 多人共享同一套菜品/食材数据

**明显不做**：支付、独立账号体系、UGC/评论、多商家/外卖

**遗留给 design 的最大未知**：食材到菜品的映射关系数据来源——是手动维护菜品-食材关系表，还是接入外部 API？这决定云数据库 Schema 设计和数据初始化策略。
